下面这份功能文档（PRD/对接规格）按「你的 SaaS = 第三方平台」视角整理，可直接拆给产品/前端/后端/测试用。  
底层依据：微信开放平台「第三方平台-快速注册企业小程序（fastregisterweapp）」机制：收集 企业名称/企业代码（含code_type）/法人姓名/法人微信号 → 调接口 → 微信向法人微信下发模板消息 → 法人 24h 内点消息做人脸核验 → 核验通过后创建已认证小程序 → 结果通过授权事件接收 URL把 appid 推给你。

1) 功能定位 & 适用边界

你要做的：在 SaaS 里给“商户/客户”提供一个引导式代注册小程序流程，让客户填最少信息，走完“主体创建 + 认证”，然后把生成的 appid 纳入你的「小程序管理」。

重要前提/约束（避免踩坑）
• 这是第三方平台能力，你们必须是审核通过的第三方平台，并配好：  

  授权事件接收 URL（Receive URL）、ticket 推送解密、component_access_token 获取链路。
• 创建任务存在串行/同主体冷却约束：官方明确提到“创建任务逻辑串行，单次任务结束后才可用相同信息下发第二次”，以及 89249 该主体已有任务执行中，距上次任务 24h 后再试等错误码。

• 不得把该流程包装成“绕过实名/人脸”：法人侧必须本人完成微信侧的人脸核身；你们绝不能替法人点消息/做核验。

2) 角色与入口

角色 视角 关键动作

SaaS 超级管理员/运营 后台 开通/配置第三方平台 Receive URL、加解密密钥；查看注册流水

SaaS 客户（商户管理员） 前台 “创建小程序”向导 → 填企业信息 → 等待法人核验 → 进入小程序管理

客户侧法人（不一定会登 SaaS） 微信侧 收到微信下发消息 → 点进去 → 补全身份证/人脸 → 成功

入口建议：  
商户设置 / 渠道管理 / 微信小程序 → 「快速创建小程序」（若已绑定 appid，改为“小程序管理”入口）

3) 核心业务流程（SaaS 端状态机）

推荐你把“代注册任务”做成一张表（例如 wx_fastregister_task），状态机如下：
[DRAFT] 草稿（未提交）
  ↓ 提交校验 & 调微信 create
[SUBMITTED] 已提交（等待法人操作）
  ↓ 微信推送“法人点击/核验中/结果”到 Receive URL
[VERIFYING] 核验中/审核中（你们可按“已收到中间事件但还未拿到appid”归为一类）
  ↓ 收到 notify 里包含 created appid/授权信息
[SUCCESS] 注册成功（已认证） → 自动落库 appid，进入【小程序管理】
[FAIL] 失败/驳回（含超时未操作、信息不一致等）


用户侧最关键的等待节点是：提交成功后 → 法人微信会收到模板消息 → 必须在 24h 内点击并完成人脸。官方明确写了“法人需在 24 小时内点击消息”。

4) 前台页面对接设计（客户要填什么、你怎么校验）

4.1 收集字段（最小集）

按微信接口必填项收敛（不要让客户填多余资质）：

字段 控件 校验/文案（给客户的提示）

企业名称 name 输入框 必须与工商登记一致（“广州XX公司”别写成“广州市XX公司”）

企业代码类型 code_type Radio/Select 强烈建议只允许 1:统一社会信用代码(18位)，最稳（18位字母数字）

企业代码 code 输入框 18位大写+数字；可跑正则兜底：^[0-9A-HJ-NP-RTUW-Y]{18}$

法定代表人姓名 legal_persona_name 输入框 提示：必须与法人微信绑定银行卡的实名姓名一致（改过名/15升18位身份证容易翻车）

法人微信号 legal_persona_wechat 输入框 提示：去微信「我-头像下方」复制微信号（不能用手机号/QQ号）

第三方联系电话 component_phone 输入框 建议预填 SaaS 客服/商户联系人手机（可选但建议）

4.2 提交前的“防呆”

• 同商户（同主体名称/代码）若处于 SUBMITTED/VERIFYING，前端+后端都锁重提，避免撞 89249

• 提交按钮点完后进“等待法人”只读页（展示法人微信脱敏：如 zhang***san），并提供：

  • 提示话术：“请让法人打开微信查看服务通知/公众号消息并点击完成核验（24h内）”

  • 自助排查入口（见 §6 失败排查）

5) 后端对接规格（最关键：调接口 + 收事件 + 解密 + 落 appid）

5.1 调用“创建小程序”接口

POST

https://api.weixin.qq.com/cgi-bin/component/fastregisterweapp?action=create&component_access_token=TOKEN

Body（JSON/Form 按官方示例，注意中文无需奇怪转码）
{
  "name": "企业全称（与工商一致）",
  "code": "9131xxxxxxxxxX",
  "code_type": 1,
  "legal_persona_wechat": "法人微信号",
  "legal_persona_name": "法人姓名",
  "component_phone": "你的SaaS客服电话"
}


常见返回 errcode 处理清单（来自官方）

errcode 含义 你的处理

0 提交成功（不代表注册完成） 写任务状态 SUBMITTED；提示“已向法人微信下发消息”

89249 该主体已有任务执行中，距上次任务 24h 后再试 回到表单锁定，提示“同一主体正在流程中，请先让法人完成核验或24h后再试”

86004 无效微信号 让客户核对“微信号”而非手机号

61070 法人姓名与微信号不一致 提示核对：微信号主人是否与银行卡实名一致/是否改名/身份证升位

89255 code参数无效（长度/内容/code_type不匹配） 回到表单强校验统一社会信用代码

89254 第三方权限集不全 你们后台检查第三方平台权限集是否缺项

-1 系统错误 指数退避重试1~2次，仍失败转人工工单

官方强调：不要频繁轮询/search；消息以服务端事件推送为准（“消息接收需通过服务器查看”“不要频繁调用 search 接口会消耗 quota”）。

5.2 授权事件接收 URL（Receive URL）怎么接（拿 appid 的唯一正路）

第三方平台的 授权事件接收 URL 会收到微信 POST（XML/加密），其中就包含“注册成功带回 appid”的事件（以及 ticket 等）。

你必须落实：
1. URL/路由只接受 POST  
2. 验签（如果有 HTTP 层签名参数）→ 解密 AESKey → 解析事件类型  
3. 对同一条 event 做幂等去重（用 ToUserName+CreateTime 或微信给的消息 id 做唯一键），避免重复落库/反复刷新状态  
4. 处理完 直接返回明文 success（或按微信约定格式），不能返回 200 JSON 却不说 success，否则可能影响重试行为

当事件表明“小程序创建成功/已授权”时：把 appid 写入你们那张任务表 + 写入商户的「小程序管理」主表，状态切 SUCCESS。

6) 失败/卡住的常见原因 & 给商户的自助排查（做成帮助文案）

把你们 FAQ 直接对齐官方列出的坑：

• 一直没收到消息  

  • 让商户确认：法人微信是否把“服务通知/公众号消息”关了  

  • 让法人搜公众号 「公众平台安全助手」 → 查“创建小程序申请”消息  

  • 第三方检查：接收微信号是否填错、是否法人本人微信号

• 提交报 61070 / 86004  

  • 是不是拿手机号当微信号？路径：「微信-我」复制微信号  

  • 法人是否改过名/身份证从15位升18位导致银行卡实名不一致

• 主体信息不一致/被驳回  

  • 企业名称一字之差（有限公司/有限责任公司/广州市vs广州）  

  • 用国家企业信用信息公示系统对照

7) 成功后 → 落到你的「小程序管理」模块（你能接着做啥）

一旦你拿到 appid，并且第三方平台拿到了该小程序的授权（authorizer_access_token），你的「小程序管理」至少可以覆盖：

1. 基本信息收纳
   • appid / 主体名 / 认证状态 / 创建时间 / 关联商户

2. 代码生命周期（你的 SaaS 价值）
   • 上传代码包（或走你的模板）→ 提审 → 发布  

   • 建议你们抽象成：SaaS 内点“一键部署/升级”，背后走第三方平台代码管理接口

3. 成员与权限
   • 绑定体验者/开发者（体验二维码）

   • 域名/业务域名配置（需要时）

4. 合规/类目
   • 提醒：快速创建也有类目约束，部分类目会被驳回（官方让参考类目参考表）
