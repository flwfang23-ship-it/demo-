## 1. 需求背景

### 1.1 两条链路，两种取链方式

```
  链路 A：客户自行取链（B05 自动取链）
  ┌─────────────────────────────────────────────────────┐
  │  商户点击[生成推广链接]                                │
  │    → 后端实时调美团/饿了么接口                         │
  │    → 每次生成全新链接，不会过期                        │
  │    → 不需要缓存                                       │
  └─────────────────────────────────────────────────────┘

  链路 B：下游功能调用（小程序 / H5）
  ┌─────────────────────────────────────────────────────┐
  │  小程序/H5 模块需要展示活动口令                        │
  │    → 不直接调平台接口（无用户交互触发）                  │
  │    → 需要提前准备好新鲜口令存在库里                     │
  │    → 读 token_cache 表获取                            │
  └─────────────────────────────────────────────────────┘
```

> **关键区分**：链路 A 是用户触发、实时生成，链接永不过期；链路 B 是系统间调用、读缓存，必须靠定时任务保证缓存中的口令不过期。

### 1.2 需要缓存的字段

美团和饿了么返回的口令类字段有效期约 30 天：

| 平台 | 字段路径 | 展示名 |
|------|----------|--------|
| 饿了么 | `taobao_promotion.tb_watchword` | 淘宝口令(短) |
| 饿了么 | `taobao_promotion.tb_watchword_suggest` | 淘宝口令(建议版) |
| 饿了么 | `taobao_promotion.scheme_url` | 唤起协议（含时效参数） |
| 美团 | `referralLinkMap["5"]` → `groupToken` | 团口令 |

### 1.3 下游消费者（后续建设）

| 模块 | 调用场景 | 需要的口令字段 |
|------|----------|---------------|
| 小程序功能 | 小程序内嵌活动页，展示可复制口令 | 全部 4 个字段 |
| H5 功能 | H5 页面展示活动口令供用户复制 | 全部 4 个字段 |

> 这些模块不做实时取链——它们从 `token_cache` 表读取。因此 `token_cache` 中的口令必须由定时任务保持新鲜。

## 3. 总后台：设置自动更新清单

### 3.1 配置入口

在总后台活动管理列表页（B02），每条联盟活动增加「自动更新口令」开关。

```
  ┌────────────────────────────────────────────────────────────┐
  │  活动管理 > 活动列表                                        │
  ├────────────────────────────────────────────────────────────┤
  │  活动名称          │供应商│状态│佣金│自动更新口令│操作      │
  ├────────────────────────────────────────────────────────────┤
  │  淘宝闪购-闪购一下  │饿了么│上架│6%  │ 🟢 开启   │[详情]   │
  │  美团外卖节        │美团 │上架│3%  │ 🟢 开启   │[详情]   │
  │  京东零食节        │京东 │下架│8%  │  —        │[详情]   │
  │  滴滴出行·新用户   │打车 │上架│5%  │  —        │[详情]   │
  └────────────────────────────────────────────────────────────┘
```

### 3.2 交互规则

| 交互 | 触发 | 说明 |
|------|------|------|
| 开启 | 点击开关 | 活动加入自动更新清单。仅美团/饿了么活动可操作，其他供应商开关灰显 |
| 关闭 | 再次点击 | 移出清单。已缓存口令不清除，但不再更新（下游仍可读最后一次缓存） |
| 首次开启 | 关→开 | **立即触发一次口令更新**，不等 1 号/15 号，确保开启后缓存立即可用 |

### 3.3 活动表新增字段

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `autoRefreshToken` | Boolean | `false` | 是否纳入自动更新清单 |

## 4. 定时任务

### 4.1 触发规则

| 项目 | 说明 |
|------|------|
| 触发时间 | 每月 1 号 03:00 + 每月 15 号 03:00 |
| 触发方式 | 服务端 Cron |
| 执行范围 | `autoRefreshToken = true` 且 `status = 'online'` 的所有联盟活动 |

### 4.2 执行流程

```
  Step 1: 查待更新活动
  SELECT * FROM activities
  WHERE autoRefreshToken = true
    AND status = 'online'
    AND supplier IN ('meituan', 'eleme')

  Step 2: 逐个活动处理
  for each activity:
    │
    ├─ 查该活动关联的所有推广位
    │   (通过活动-推广位关联表)
    │
    ├─ 对每个推广位：
    │   ├─ 取该推广位的 platformConfig
    │   ├─ 调平台取链接口 (传入 platformConfig + activityId)
    │   ├─ 提取口令字段：
    │   │   美团: referralLinkMap["5"]
    │   │   饿了么: taobao_promotion.tb_watchword
    │   │          taobao_promotion.tb_watchword_suggest
    │   │          taobao_promotion.scheme_url
    │   └─ UPSERT 写入 token_cache
    │
    └─ 记录结果

  Step 3: 汇总日志 + 异常告警
```

### 4.3 提取逻辑

```python
def extract_token_fields(supplier, platform_resp):
    """从平台返回中仅提取需要缓存的口令字段"""
    if supplier == 'meituan':
        rlm = platform_resp.get('referralLinkMap', {})
        return {
            'groupToken': rlm.get('5', '')
        }

    elif supplier == 'eleme':
        tb = platform_resp.get('taobao_promotion', {})
        return {
            'tbWatchword': tb.get('tb_watchword', ''),
            'tbWatchwordSuggest': tb.get('tb_watchword_suggest', ''),
            'schemeUrl': tb.get('scheme_url', '')
        }
```

### 4.4 异常处理

| 场景 | 处理 |
|------|------|
| 单个活动取链失败 | 记录日志，继续处理下一个。`updateStatus` 标记为 `failed` |
| 平台接口超时 | 重试 1 次，仍失败则跳过 + 告警 |
| 推广位 platformConfig 为空 | 跳过该推广位，记录"配置缺失" |
| 返回中口令字段为 null | 写入空字符串（下游自行判断） |
| 定时任务整体失败 | 飞书/钉钉告警 |

## 5. 缓存表设计

### 5.1 表结构

```sql
CREATE TABLE activity_token_cache (
    id                 BIGINT AUTO_INCREMENT PRIMARY KEY,
    activity_id        VARCHAR(32)  NOT NULL COMMENT '活动ID',
    promotion_slot_id  VARCHAR(32)  NOT NULL COMMENT '推广位ID',
    supplier           VARCHAR(16)  NOT NULL COMMENT '平台: meituan / eleme',
    token_data         JSON         NOT NULL COMMENT '口令字段键值对',
    updated_at         DATETIME     NOT NULL COMMENT '最后更新时间',
    update_status      VARCHAR(16)  NOT NULL DEFAULT 'success' COMMENT 'success / failed',
    created_at         DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,

    UNIQUE KEY uk_activity_slot (activity_id, promotion_slot_id),
    INDEX idx_supplier (supplier),
    INDEX idx_updated_at (updated_at)
) COMMENT '口令自动更新缓存表，供小程序/H5等下游模块读取';
```

### 5.2 tokenData 示例

**美团**：
```json
{
  "groupToken": "1来美团，吃得更好，生活更好❤️复制整条信息，打开👉美团👈 http://..."
}
```

**饿了么**：
```json
{
  "tbWatchword": "￥nNqo5DUaUHe￥/ HU7405",
  "tbWatchwordSuggest": "67 HU7405  2:/£nNqo5DUaUHe《三闪购大额满减红包",
  "schemeUrl": "tbopen://m.taobao.com/tbopen/index.html?..."
}
```

## 6. 下游读取方式

小程序功能和 H5 功能通过以下方式读取口令：

```sql
-- 按活动+推广位查询
SELECT token_data, updated_at
FROM activity_token_cache
WHERE activity_id = ? AND promotion_slot_id = ?
  AND update_status = 'success';
```

| 场景 | 返回 | 下游处理 |
|------|------|----------|
| 命中 + `updatedAt` 在 15 天内 | 返回 tokenData | 正常使用 |
| 命中 + `updatedAt` 超过 15 天 | 返回 tokenData + 标记 `stale: true` | 下游自行决定是否展示"口令可能过期"提示 |
| 未命中 | 返回空 | 下游展示"口令暂不可用"或隐藏该入口 |
| `updateStatus = 'failed'` | 返回空 | 同上 |

> 下游模块**只读不写**，不对 token_cache 做任何修改。

## 7. 时效性保障

```
  1号          15号         30号(下月1号)
  │            │            │
  ├─ 更新 ──── ├─ 更新 ──── ├─ 更新
  │            │            │
  │← 14天 →│← 15天 →│← 15天 →│
  │            │            │
  任何时候读缓存，口令最旧 16 天
  口令有效期 30 天 → 安全余量 14 天 ✓
```

两次定时更新之间最长间隔 16 天（15 号→次月 1 号），远小于 30 天有效期。

## 8. 首次开启活动的处理

运营对一个活动首次开启「自动更新口令」时：

```
  开关 关→开
      │
      ▼
  后端同步执行一次口令更新
      │
      ├─ 成功 → 写入 token_cache → 开关保持 🟢
      └─ 失败 → Toast 提示 → 开关回弹 ⚪
```

> 确保不会出现"已开启但缓存为空、下游模块读不到数据"的情况。

## 9. 告警

| 监控项 | 触发条件 | 方式 |
|--------|----------|------|
| 定时任务整体失败 | Cron 未执行或异常中断 | 飞书/钉钉通知 |
| 单个活动连续失败 | 连续 2 次定时任务均 `updateStatus=failed` | 标记 + 通知运营 |
| 缓存过期 | 取链时发现 `updatedAt` > 15 天 | warn 日志 |

## 10. B05（客户取链）与本模块的关系

| | B05 客户取链 | B05-A 口令自动更新 |
|------|-------------|-------------------|
| 触发方 | 商户点击[生成推广链接] | 定时任务（1号/15号） |
| 是否调平台接口 | ✅ 每次实时调用 | ✅ 每次实时调用 |
| 是否读 token_cache | ❌ 不读 | — |
| 是否写 token_cache | ❌ 不写 | ✅ 写入 |
| token_cache 消费者 | — | 小程序功能、H5 功能 |

> **两个模块独立运行，互不干扰。** B05 客户取链不经过缓存，每次生成全新链接。

## 11. 待确认事项

1. **下游模块的具体字段需求**：小程序/H5 功能是否只用这 4 个口令字段？还是也需要其他字段（如二维码图片）？如果需要更多字段，token_cache 的提取范围要扩大。
2. **活动与推广位的关联关系**：一个活动关联多个推广位时，是否需要为每个推广位分别缓存？还是取其中一个即可？当前设计按 `activityId + slotId` 分别缓存。
3. **定时任务时间**：1 号/15 号 03:00 是否合适？
4. **缓存过期后的清理**：已关闭自动更新的活动，缓存保留多久？建议 30 天后清理。
5. **下游模块对 `stale` 的处理**：当下游读到超过 15 天的缓存时，是隐藏入口还是展示过期提示？
