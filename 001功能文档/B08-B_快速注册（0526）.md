# 快速注册通道：B端功能模块规格

> 本文档按 G 模板（B端）结构编写，覆盖商户通过 SaaS 平台代注册企业小程序（微信第三方平台 fastregisterweapp 机制）的完整流程。商户填写企业信息 → 微信向法人下发模板消息 → 法人 24h 内完成人脸核验 → 获得已认证的正式小程序 appid → 自动纳入 B08 小程序管理。

---

## 模块概览

| 项目 | 内容 |
|------|------|
| 模块名称 | 快速注册小程序（企业法人认证通道） |
| 所属层级 | 🏪 客户端后台（主要操作层）+ 🏢 总后台（查看注册流水） |
| 功能入口 | 🏪 客户端：小程序管理 → [+ 入驻小程序] → 「快速注册」<br>🏪 客户端：新人指引 Step3 → 「快速注册小程序」快捷入口 |
| 覆盖功能 | B08-B |
| 覆盖页面 | fastregister-form（注册表单页）、fastregister-waiting（等待法人核验页） |
| 时间 | 2026-05-26 |
| 依赖模块 | B01 用户登录注册（商户需已注册登录）、B08 小程序管理（appid 纳管目标模块）、微信第三方平台（component_access_token、授权事件接收 URL、AES 加解密） |

> 系统预置规则：快速注册功能依赖微信第三方平台的「快速注册企业小程序」权限集，需在微信开放平台提前开通。必须提前配置好 component_appid、component_secret、AES 加解密密钥、授权事件接收 URL（Receive URL）并验证通过，否则注册流程无法走通。

## 功能 B08-B：快速注册通道

---

### 1. 需求背景与目标 (Why)

商户需要正式的小程序来开展业务，但传统注册流程需要在微信公众平台自行操作，路径长、门槛高。SaaS 作为第三方平台，通过微信「快速注册企业小程序（fastregisterweapp）」机制，代商户收集最少的企业信息（名称+代码+法人姓名+法人微信号），提交后微信向法人下发模板消息，法人 24h 内完成人脸核验即可获得一个**已认证**的正式小程序，无需商户自行操作微信公众平台。

核心目标：
- **极简填写**：最少 4 个必填字段（企业名称 + 信用代码 + 法人姓名 + 法人微信号），不需要营业执照上传
- **自动化认证**：微信侧自动完成主体认证，无需额外提交认证材料/认证费
- **直接可用**：注册成功即为已认证正式小程序，可直接进入代码上传/提审/发布流程
- **全程可视化**：商户可实时查看注册进度（等待法人 → 核验中 → 成功/失败）

```

      │
```

```
            │   · 法定代表人姓名
            │   · 法人微信号（非手机号/QQ号）
            │   · 联系电话（可选）
            │
            ├── 前端防呆校验：
            │   · 信用代码格式：18位大写字母+数字

            │     │
            │     ▼

            │     │
            │     ├── 成功 → SUBMITTED
            │     │     │
            │     │     ▼
            │     │   进入等待法人核验页（fastregister-waiting）
            │     │
            │     ├── errcode=89249 → "该主体已有任务执行中，请24小时后重试"
            │     ├── errcode=86004 → "无效微信号，请核对"
            │     ├── errcode=61070 → "法人姓名与微信号不一致"
            │     └── errcode=89255 → "信用代码参数无效"
            │
            ▼
      Step B：等待法人核验（fastregister-waiting）
            │
            ├── 展示状态：已提交，等待法人操作
            │
            ├── 法人微信脱敏展示：如 zhang***san
            │
            ├── 提示话术：
            │   · "请让法人打开微信查看「服务通知」消息并点击完成核验"
            │   · "法人需在 24 小时内完成人脸核验"
            │
            ├── 轮询/等待微信回调
            │     │
            │     ├── 收到"法人点击消息"→ 状态更新为 VERIFYING
            │     ├── 收到"核验通过+appid" → 状态更新为 SUCCESS
            │     │     │
            │     │     ▼
            │     │   appid 自动写入 tenant_miniapp 表
            │     │   → 进入「小程序管理」（正式小程序，已认证）
            │     │   → 页面提示"注册成功"并引导进入小程序详情
            │     │
            │     └── 24h 超时未操作 / 核验驳回 → FAIL
            │           │
            │           ▼
            │   展示失败原因 + "重新发起注册"按钮
            │
            └── 自助排查入口（FAQ 帮助文案）
                  ├── 一直没收到消息怎么办
                  ├── 报 61070 / 86004 怎么处理
                  └── 主体信息不一致怎么核对
```

| 步骤 | 角色 | 触发动作 | 所在页面 | 前置条件 | 后续页面 |
|------|------|----------|----------|----------|----------|
| 1 | 🏪 商户 | 选择"快速注册" | 入驻方式弹窗 | 已登录 | fastregister-form |
| 2 | 🏪 商户 | 填写信息并提交 | fastregister-form | 校验通过 | fastregister-waiting 或 错误提示 |
| 3 | 🏪 法人 | 微信收到模板消息 | 微信端 | 注册任务提交成功 | 微信人脸核验页 |
| 4 | 🏪 法人 | 完成人脸核验 | 微信端 | 法人身份验证通过 | - |
| 5 | 🏪 商户 | 收到注册成功通知 | fastregister-waiting | 微信推送 appid | 跳转小程序详情 |
| 6 | 🏪 商户 | 注册失败，重新发起 | fastregister-waiting | 状态为 FAIL | fastregister-form（编辑模式） |

#### 2.3 快速注册状态流转图

```
商户点击"快速注册"
      │
      ▼
┌──────────────┐
│    DRAFT     │  草稿（填写中，未提交）
│   （草稿）    │
└──────┬───────┘
       │ 点击"提交注册" + 前端校验通过
       │ 后端调微信 fastregisterweapp?action=create
       ▼
┌──────────────┐
│  SUBMITTED   │  已提交（微信收到请求，等待法人操作）
│  （已提交）    │  └── 法人微信收到模板消息（24h 内有效）
└──────┬───────┘
       │
       ├── 微信推送"法人已点击消息/进入核验"
       ▼
┌──────────────┐
│  VERIFYING   │  核验中（法人正在进行人脸核验）
│  （核验中）    │
└──────┬───────┘
       │
   ┌───┼───┐
   │   │   │
   │   │   └── 24h 超时未操作 / 法人拒签 / 信息不一致
   │   │         ▼
   │   │   ┌──────────────┐
   │   │   │    FAIL      │  失败/驳回
   │   │   │  （注册失败）  │  └── 展示失败原因
   │   │   └──────┬───────┘     帮助文案 + 重新发起
   │   │          │
   │   │          └── 回到 DRAFT（重新填写并提交）
   │   │
   │   └── 微信推送"核验通过 + 已创建小程序 appid"
   │         ▼
   │   ┌──────────────┐
   │   │   SUCCESS    │  注册成功
   │   │ （注册成功）   │  └── appid 写入 tenant_miniapp
   │   └──────────────┘      sourceType = official
   │                         verified = true
   │                         → 进入 B08 小程序管理（正式）
```

**状态说明表**：

| 状态名称 | 状态含义 | 进入条件 | 可触达的状态 | 操作角色 |
|----------|----------|----------|-------------|---------|
| DRAFT | 草稿（未提交） | 商户进入表单开始填写 | SUBMITTED | 🏪 商户 |
| SUBMITTED | 已提交，等待法人操作 | 微信 create 接口返回成功（errcode=0） | VERIFYING, FAIL | 🏪 法人（微信侧） |
| VERIFYING | 法人核验中 | 微信推送中间事件（法人已点击/进入核验） | SUCCESS, FAIL | 微信侧 |
| SUCCESS | 注册成功 | 微信推送核验通过 + 已创建 appid | - | 系统自动 |
| FAIL | 注册失败/驳回 | 超时未操作 / 法人拒签 / 信息不一致 / 微信驳回 | DRAFT（重新发起） | 🏪 商户 |

#### 4.3 帮助文案（FAQ，直接展示在页面中）

##### 4.3.1 一直没收到消息怎么办

```
1. 请确认法人微信是否关闭了「服务通知」或公众号消息
2. 让法人搜索公众号 「公众平台安全助手」→ 查看"创建小程序申请"消息
3. 检查填写的法人微信号是否正确（去微信「我」复制微信号，不是手机号）
4. 确认微信号主人是否为法人本人
```

##### 4.3.2 提交报"法人姓名与微信号不一致"（61070）

```
1. 确认填写的微信号是否为手机号（手机号无效，需要用微信号）
2. 法人是否改过名字？改名后的姓名需与银行卡实名一致
3. 法人身份证是否从15位升级到18位？升位后需确保银行卡实名已更新
```

##### 4.3.3 提交报"无效微信号"（86004）

```
1. 确认填写的不是手机号/QQ号
2. 正确路径：微信「我」→ 点击头像 → 「微信号」一栏的内容
3. 微信号通常为字母+数字组合，不是纯数字
```

##### 4.3.4 主体信息不一致/被驳回

```
1. 企业名称是否一字不差？（"有限公司"≠"有限责任公司"、"广州市"≠"广州"）
2. 用国家企业信用信息公示系统（gsxt.gov.cn）核对最新工商登记信息
3. 确认统一社会信用代码是否为18位且无误
```

**后端处理流程**：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "taskId": 1001,

  }

```

```json
{
  "code": 0,
  "message": "success",

    "taskId": 1001,
    "status": "FAIL",
    "wxErrorCode": 89249,
    "wxErrorMessage": "该主体已有任务执行中，距上次任务24小时后再试",
```

| 错误码 | 说明 | 前端处理 |
|--------|------|----------|
| 89249 | 该主体已有任务执行中 | 表单锁定，提示"24小时后重试" |
| 86004 | 无效微信号 | 法人微信号字段标红 |

**API-CB001：微信授权事件回调**

**调用场景**：

| 项目 | 说明 |
|------|------|
| 适用层级 | 后端（无前端调用） |
| 触发时机 | 微信向 Receive URL POST 事件 |
| 调用频率 | 微信推送频率 |

**处理逻辑**：
1. 验签（HTTP 层签名参数）
2. AES 解密 XML 消息体
3. 解析事件类型（InfoType）
4. 幂等去重（用 ToUserName + CreateTime 或消息 ID 做唯一键）
5. 根据事件类型更新对应任务的 status：
   - 法人点击/核验中 → VERIFYING
   - 核验通过 + 已创建 appid → SUCCESS + 写入 tenant_miniapp
   - 核验驳回/失败 → FAIL + 记录失败原因
6. 返回明文 "success"

#### 5.3 业务规则

| 规则编号 | 规则描述 | 适用层级 | 触发条件 | 处理逻辑 | 异常处理 |
|----------|----------|----------|----------|----------|----------|
| RULE-FR-001 | 同一主体串行限制 | 🏪 客户端 | 提交注册 | 检查同主体（名称+代码）是否有 SUBMITTED/VERIFYING 的任务 | 前端+后端双重锁，返回 89249 类错误 |
| RULE-FR-002 | 法人 24h 操作时限 | 🏪 客户端 | 任务 SUBMITTED 后 | 24h 倒计时，超过未操作 → FAIL | 提示"法人超时未操作" |
| RULE-FR-003 | 代码类型仅支持1 | 🏪 客户端 | 填写代码 | 仅允许 codeType=1（统一社会信用代码），前端下拉默认选中且不可切换 | 灰度提示 |
| RULE-FR-004 | 不要频繁轮询 search 接口 | 后端 | - | 不要调用微信 search 接口轮询状态，以 Receive URL 事件推送为准 | 消耗 quota |
| RULE-FR-005 | 幂等去重 | 后端 | 收到微信事件 | 以唯一键做幂等去重，防止重复落库 | - |
| RULE-FR-006 | 回调返回 success | 后端 | 处理完微信事件 | 返回明文 "success"，不能返回 200 JSON 却不是 success | 否则影响微信重试行为 |
| RULE-FR-007 | 注册成功后自动纳管 | 后端 | 状态变为 SUCCESS | appid + 主体信息写入 tenant_miniapp 表，sourceType=official, verified=true | 记录操作日志 |
| RULE-FR-008 | 注册失败可重试 | 🏪 客户端 | 状态为 FAIL | 允许重新发起，注意 24h 冷却期 | 提示冷却时间 |

## 模块汇总

**关键设计决策**：
1. 快速注册作为"正式入驻"通道，面向已有企业主体、希望直接获得正式小程序的商户
2. 收集字段最小化（4 必填 + 1 选填），不需要上传营业执照（微信侧通过法人扫脸完成认证）
3. 不轮询微信 search 接口（消耗 quota），完全依赖 Receive URL 事件推送获取状态变更
4. 法人 24h 操作时限 + 同一主体串行限制，前端+后端双重防呆
5. 注册成功后自动写入 tenant_miniapp 表，无缝进入小程序管理
6. 法人不能代操作是硬约束（合规要求），SaaS 绝不能替法人点消息/做核验
