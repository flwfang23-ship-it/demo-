> **核心设计原则**：收入归因是推广结算的基础。系统根据用户进入的端（小程序 vs 直推H5）采用不同的归因策略——小程序端基于"用户标签"持久化追踪，直推H5端基于"页面属性"即时归因。商户无需手动配置归因规则，系统在取链时自动嵌入推广位标识，在用户产生收入时自动完成归因。

> **⚠️ 本文档范围说明**：本文档覆盖收入归因的完整业务规则——小程序端四类场景（自然流量、推广流量、异常情况、分享裂变）和直推H5端归因逻辑。归因规则的配置界面（如归因窗口期、标签过期策略等）归属后续 B03-C 模块，本文档仅描述核心归因逻辑。

## 小程序端收入归因

---

### 1. 需求背景与目标 (Why)

商户通过 B03-A 取链生成推广链接后投放至微信生态，用户点击链接进入小程序并产生消费。系统需要精准识别"这个用户是经由哪个推广位带来的"，并将该用户产生的收入准确归因到对应推广位，作为推广人员的业绩结算依据。

小程序端的核心挑战在于：用户可能多次进入小程序（不总是通过推广链接），也可能将小程序分享给他人。因此小程序端采用**用户标签机制**——给用户打上推广位标签，标签具有记忆性、可覆盖性和可传递性，确保归因的连续性和准确性。

---

### 2. 业务流程与逻辑 (How - 宏观)

#### 2.1 归因模型总览

```
  B03-A 取链（链接携带推广位标识）
        │
        │ 用户点击推广链接
        ▼
  ┌──────────────────────────────────────────────┐
  │              判断进入端类型                     │
  ├──────────────────────────────────────────────┤
  │  小程序端                    直推H5端          │
  │  （基于用户标签）     →     （基于页面属性）     │
  │  标签持久化、可传递            即时绑定、不持久化 │
  └──────────────────────────────────────────────┘
        │                              │
        │ 用户产生收入                  │ 用户产生收入
        ▼                              ▼
  查用户标签 → 归因到标签推广位    查页面推广位参数 → 归因到页面推广位
```

#### 2.2 小程序端归因流程图

```
  用户进入小程序
        │
        ├── 是否通过推广链接进入？
        │       │
        │       ├── 是 → 从链接解析推广位标识
        │       │         │
        │       │         ├── 推广位有效（未被删除）→ 给用户打上/更新"推广位X"标签
        │       │         └── 推广位已被商户删除 → 不打标签 / 清除旧标签
        │       │
        │       └── 否 → 进入方式判定
        │                   │
        │                   ├── 自然流量（搜索/直接访问）
        │                   │   └── 不打标签，用户无标签
        │                   │
        │                   └── 分享进入（通过其他用户分享）
        │                       └── 检查分享者标签
        │                           ├── 分享者有标签"推广位A" → 新用户继承"推广位A"标签
        │                           └── 分享者无标签 → 新用户无标签
        │
        ▼
  用户产生收入时
        │
        ├── 用户有标签"推广位X"
        │   └── 检查推广位X是否有效
        │       ├── 有效 → 收入归因到"推广位X"
        │       └── 已被商户删除 → 收入归因到"默认推广位"
        │
        └── 用户无标签
            └── 收入归因到"默认推广位"
```

## 直推H5端收入归因

---

### 1. 需求背景与目标 (Why)

商户通过 B03-A 取链生成 H5 推广链接后，用户直接访问该 H5 页面进行浏览和转化。与小程序端不同，H5 端的用户行为不依赖微信生态的用户标识体系，因此采用更直接的**页面属性归因**——收入直接绑定到当前 H5 页面所携带的推广位参数。

直推H5端归因的核心特点是**即时性**和**无状态性**：不依赖用户历史标签，不跨页面传递，谁的地盘消费，业绩就归谁。

---

### 2. 业务流程与逻辑 (How - 宏观)

#### 2.1 直推H5端归因流程图

```
  用户访问 H5 页面
        │
        │ URL 携带推广位参数（如 ?slotId=PROMO_006）
        ▼
  系统解析推广位标识
        │
        ├── 推广位参数有效 → 页面绑定"推广位C"
        └── 推广位参数缺失/无效 → 页面绑定"默认推广位"
        │
        ▼
  用户在 H5 页面产生收入（支付完成）
        │
        ▼
  收入归因到该 H5 页面当前绑定的推广位
  （不考虑用户历史标签、不跨页面传递）
```

#### 2.2 与小程序端的关键差异

| 维度 | 小程序端 | 直推H5端 |
|------|---------|---------|
| 归因依据 | 用户标签（持久化） | 页面参数（即时） |
| 标签/参数生命周期 | 长期有效，直到被覆盖或推广位被商户删除 | 仅当前页面会话有效 |
| 跨会话保持 | 是（标签跟随用户） | 否（每次访问重新判断） |
| 分享传递 | 是（新用户继承分享者标签） | 否（分享后按新页面参数归因） |
| 多次点击不同链接 | 标签被覆盖为最新链接的推广位 | 每次按当前页面参数独立归因 |
| 推广位被商户删除后 | 回退归因到默认推广位 | 页面参数失效，归因到默认推广位 |

## 跨层通用

### 5.1 归因规则详述

#### 5.1.1 场景一：自然流量（无推广介入）

| 项目 | 说明 |
|------|------|
| 触发场景 | 用户通过主动搜索、浏览器收藏夹、直接输入 URL 等自然方式进入小程序 |
| 端类型 | 仅小程序端（H5端不存在"自然流量"概念，每次访问必然携带页面参数） |
| 处理逻辑 | 系统不给用户打任何推广位标签 |
| 收入归因 | 该用户产生的收入，归因到**"默认推广位"** |
| 特殊说明 | 若用户此前已有标签（曾通过推广链接进入过），则保留原标签，收入仍归因到原推广位。仅在用户从未有过标签时，才归因到默认推广位 |

#### 5.1.2 场景二：推广流量（通过运营链接进入）

| 项目 | 说明 |
|------|------|
| 触发场景 | 用户点击商户通过 B03-A 生成的特定推广链接（携带推广位标识参数）进入小程序 |
| 端类型 | 小程序端 + 直推H5端 |
| 处理逻辑 | **小程序端**：系统从链接参数中解析推广位标识，给该用户打上对应推广位标签。此标签会持续跟随用户，后续用户无论从小程序哪个入口进入（首页、活动页等），都视为带有此标签。**直推H5端**：系统从 URL 参数解析推广位标识，绑定到当前页面会话。 |
| 收入归因 | **小程序端**：无论在小程序内何处产生交易，收入均归因到该推广位。**直推H5端**：用户在绑定该推广位的 H5 页面上完成支付，收入归因到该推广位。 |
| 覆盖规则 | **小程序端**：若用户已有旧标签，新链接的推广位标识会**覆盖**旧标签。后续收入归因到新推广位。**直推H5端**：每次页面访问独立判断，不存在覆盖。 |

#### 5.1.3 场景三：异常情况（标签变更或失效）

| 异常类型 | 触发条件 | 处理逻辑 | 收入归因 |
|----------|----------|----------|----------|
| 推广位被商户删除 | 商户在 B03 推广位列表中删除了推广位 A（客户端操作，非总后台操作） | 原带有 A 标签的用户，在收入产生时检测到推广位 A 已不存在 | 收入回退归因到**"默认推广位"**，同时清除用户的 A 标签 |
| 接受新推广链接 | 已带标签的用户再次点击了新的推广链接（推广位 B） | 系统将用户标签更新/覆盖为 B | 后续产生的收入归因到推广位 B |
| 链接参数异常 | 推广链接中的推广位标识无法解析或不存在 | 不打标签/不绑定推广位 | 小程序端：保留用户现有标签（如有）或归因到默认；H5端：归因到默认推广位 |

#### 5.1.4 场景四：分享流量（属性传递 / 社交裂变）

| 项目 | 说明 |
|------|------|
| 触发场景 | 带有推广位标签的老用户（如带有"推广位 A"标签的用户 Q）将小程序分享给新用户 O |
| 端类型 | 仅小程序端（H5端分享出去的链接自带推广位参数，按场景二处理） |
| 处理逻辑 | 发生**属性继承**。新用户 O 进入小程序后，系统检测到其通过分享链接进入，查询分享者 Q 的用户标签，将 Q 的推广位标签复制给 O |
| 收入归因 | 新用户 O 产生的收入与分享者保持一致，归因到"推广位 A" |
| 多层分享 | 若 O 再分享给 P，P 同样继承 O 的标签（即原始推广位 A），形成裂变链。裂变链上所有用户的收入均归因到最初推广位 |
| 分享者无标签 | 若分享者本身无标签（自然流量用户），新用户也不打标签，收入归因到默认推广位 |

---

### 5.2 归因参数编码规则

B03-A 取链时，系统将推广位标识编码到链接参数中。

#### 5.2.1 小程序推广链接参数

| 参数名 | 类型 | 必传 | 说明 |
|--------|------|------|------|
| `slotId` | String | 是 | 推广位ID，如 PROMO_006 |
| `s` | String | 否 | 分享者标识（分享裂变时携带，用于追溯分享链） |
| `t` | Number | 否 | 链接生成时间戳（用于归因窗口期判断） |

**示例**：
```
/pages/benefit/index?slotId=PROMO_006&t=1716710400
```

#### 5.2.2 直推H5链接参数

| 参数名 | 类型 | 必传 | 说明 |
|--------|------|------|------|
| `slotId` | String | 是 | 推广位ID，如 PROMO_006 |
| `t` | Number | 否 | 链接生成时间戳 |

**示例**：
```
https://h5.xxx.com/benefit?slotId=PROMO_006&t=1716710400
```

### 5.3 数据流

| 数据项 | 来源 | 更新时机 | 数据范围（总后台） | 数据范围（客户端） | 缓存策略 |
|--------|------|----------|--------------------|--------------------|----------|
| 用户标签（小程序端） | 用户点击推广链接时写入 | 每次点击推广链接 / 分享裂变 / 推广位被商户删除 | 全部商户全部用户 | —（不可见） | Redis 缓存，标签变更实时同步 |
| 页面绑定（H5端） | 页面加载时从URL解析 | 每次页面加载 | — | — | 会话级，不持久化 |
| 归因记录 | 支付完成时生成 | 每次支付 | 全部商户 | 当前商户 | 无缓存 |
| 归因概览统计 | 定时汇总 | 每小时更新 | 全部商户 | 当前商户 | 1小时缓存 |
| 推广位状态 | B03 推广位管理 | 商户操作时变更 | 全部商户 | 当前商户 | 无缓存 |
| 默认推广位 | 商户注册时创建 | 注册时 | 全部商户 | 当前商户 | 无缓存 |

### 5.5 业务规则

| 规则编号 | 规则描述 | 适用层级 | 触发条件 | 处理逻辑 | 异常处理 |
|----------|----------|----------|----------|----------|----------|
| RULE-B03B-001 | 小程序端标签持久化 | 🏢 + 🏪 | 用户通过推广链接进入小程序 | 系统从链接参数解析推广位ID → 写入用户标签（Redis）→ 标签长期有效 | 写入失败重试3次，仍失败则本次归因到默认推广位 |
| RULE-B03B-002 | 标签覆盖规则 | 🏢 + 🏪 | 已带标签的用户点击新推广链接 | 新链接的推广位ID覆盖旧标签 → 后续收入归因到新推广位 | 覆盖为原子操作，避免并发覆盖导致标签错乱 |
| RULE-B03B-003 | 推广位删除回退 | 🏢 + 🏪 | 推广位被商户在B03删除，且带该标签的用户产生收入 | 收入结算时检测推广位是否存在 → 若已被删除 → 归因到默认推广位 + 清除用户标签 | — |
| RULE-B03B-004 | 分享属性继承 | 🏢 + 🏪 | 带标签用户分享小程序给新用户 | 新用户继承分享者的推广位标签 → 收入与分享者归因一致 | 分享者无标签时，新用户不打标签，归因到默认 |
| RULE-B03B-005 | 多层分享链追溯 | 🏢 + 🏪 | 分享裂变超过1层 | 裂变链逐层传递原始推广位标签，最深追溯至原始推广链接点击者 | 裂变链超过10层仍保留标签，但前端展示仅显示最近5层 |
| RULE-B03B-006 | 直推H5端无标签 | 🏢 + 🏪 | 用户访问H5页面 | H5端不写用户标签，仅从URL参数绑定推广位到当前页面会话 → 收入按页面绑定即时归因 | URL参数缺失时归因到默认推广位 |
| RULE-B03B-007 | H5端无分享继承 | 🏢 + 🏪 | 用户分享H5页面 | H5页面分享时，分享链接自带页面推广位参数。新用户访问时按URL参数独立归因，不查分享者历史 | — |
| RULE-B03B-008 | 归因到默认推广位 | 🏢 + 🏪 | 用户无标签 或 标签推广位无效 或 链接参数缺失 | 收入统一归因到该商户的默认推广位 | 若默认推广位也不存在（异常情况），归因到系统级兜底账户，触发告警 |
| RULE-B03B-009 | 归因窗口期 | 🏢 + 🏪 | 用户点击推广链接后N天内产生收入 | 归因窗口期内（默认30天可配置），收入归因到推广链接对应的推广位；超过窗口期，若用户无后续推广链接点击，归因到默认推广位 | 窗口期配置由总后台统一管理（B03-C） |
| RULE-B03B-010 | 同一支付不重复归因 | 🏢 + 🏪 | 一笔支付同时满足多个归因条件 | 优先级：推广链接直达 > 分享裂变 > H5页面参数 > 自然流量（默认推广位）。每笔收入仅归因一次 | — |

## 模块汇总

### 关键设计决策

1. **小程序端 vs H5端双轨归因**：小程序端因微信生态内用户标识稳定，采用"用户标签"持久化追踪，支持跨会话、跨入口的持续归因；直推H5端因用户标识不稳定，采用"页面属性"即时归因，简单直接。

2. **标签三性设计**：用户标签具备记忆性（持久跟随用户）、可覆盖性（新链接覆盖旧标签）、可传递性（分享继承），覆盖了自然流量、推广流量、分享裂变三类核心场景。

3. **推广位删除保护**：推广位被商户在客户端删除后，已打标签的用户收入自动回退到默认推广位，防止收入归因到无效推广位导致结算异常。

4. **分享裂变链追溯**：支持多层分享的属性继承和追溯，裂变链上所有用户的收入均归因到最初推广位，激励商户和推广人员的分享传播行为。

5. **归因优先级明确**：推广链接直达 > 分享裂变 > H5页面参数 > 自然流量（默认推广位），确保同一笔收入不重复归因。

6. **商户端简化展示**：商户无需理解底层归因逻辑的复杂性（小程序 vs H5、标签 vs 页面参数），客户端仅展示归因结果（收入总额 + 订单数），降低认知负担。

### 待办/注意事项

- 归因窗口期的默认值（30天）是否需要在客户端可见？建议在 B03-C 配置模块中增加商户可见的窗口期展示。
- 用户标签的存储方案：Redis 的持久化策略（AOF/RDB）需确认，防止标签数据丢失导致大规模归因回退到默认推广位。
- 分享裂变链的存储：每条裂变关系是否需要独立存储？当前设计为支付时实时查询分享链。
- 高并发下的标签覆盖：同一用户短时间内点击多个推广链接时，标签覆盖的并发控制策略需后端确认（乐观锁 vs 分布式锁）。
- 归因数据的修正能力：若因系统故障导致归因错误（如推广位状态未及时同步），是否需要支持人工修正归因记录？如需，应在 B03-C 中增加"归因修正"功能。
