## 功能 A001：第三方活动分组与预设活动管理（🏢 总后台）

---

### 1. 需求背景与目标 (Why)

平台运营方需要统一管理第三方活动的基础数据：分组归类所有第三方活动，预设活动作为全商户共用的活动模板。商户在客户端可以看到并引用这些预设活动，无需自己从零建立。

---

### 2. 业务流程与逻辑 (How - 宏观)

#### 2.1 总后台操作路径（🏢）

```
  进入分组列表页        点击[编辑]分组         修改名称/排序
  (活动管理→第三方分组) ───► 编辑弹窗 ───► 保存 → 列表刷新
       │
       │ 点击分组行
       ▼
  进入预设活动列表       点击[+ 新增活动]      填写活动信息
  (该分组下的活动)  ───► 编辑弹窗        ───► 保存 → 列表刷新
       │                      │
       │ 点击[编辑]活动       │ 点击[删除]活动
       ▼                      ▼
  编辑弹窗（名称/AppID）   二次确认 → 删除 → 列表刷新
```

| 步骤 | 角色 | 触发动作 | 所在页面 | 前置条件 | 后续页面 |
|------|------|----------|----------|----------|----------|
| 1 | 🏢 运营 | 进入第三方活动分组列表 | 页面 B04 | 已登录运营账号 | - |
| 2 | 🏢 运营 | 点击[编辑]某分组 | 页面 B04 | 分组存在 | 弹窗 D-B04-A |
| 3 | 🏢 运营 | 修改分组名/排序并保存 | 弹窗 D-B04-A | 名称不重复 | 列表刷新 |
| 4 | 🏢 运营 | 点击[新增分组] | 页面 B04 | - | 弹窗 D-B04-A |
| 5 | 🏢 运营 | 点击分组行 | 页面 B04 | 分组存在 | 页面 B04-B（预设活动列表） |
| 6 | 🏢 运营 | 点击[+ 新增活动] | 页面 B04-B | - | 弹窗 D-B04-C |
| 7 | 🏢 运营 | 填写活动名/AppID并保存 | 弹窗 D-B04-C | 必填项校验通过 | 列表刷新 |
| 8 | 🏢 运营 | 点击[删除]预设活动 | 页面 B04-B | 活动存在 | 二次确认 → Toast + 列表刷新 |

#### 2.2 状态流转图

```
  ┌──────────────┐   运营[新增分组]     ┌──────────────┐
  │   无分组      │──────────────────►│  分组已启用    │
  │   (初始态)   │                    │  (enabled)   │
  └──────────────┘                    └──────┬───────┘
                                             │
                                             │ 运营[禁用]/[删除]
                                             ▼
                                        ┌──────────────┐
                                        │  分组已禁用    │
                                        │  (disabled)  │
                                        └──────────────┘

  预设活动状态流转：

  ┌──────────────┐   运营[新增活动]     ┌──────────────┐
  │   无活动      │──────────────────►│  已发布       │
  │   (初始态)   │                    │ (客户端可见)  │
  └──────────────┘                    └──────┬───────┘
                                             │
                                             │ 运营[删除]
                                             ▼
                                        ┌──────────────┐
                                        │  已删除       │
                                        │ (客户端不可见)│
                                        └──────────────┘
```

| 状态名称 | 状态含义 | 进入条件 | 可触达的状态 | 操作角色 |
|----------|----------|----------|-------------|---------|
| 分组-已启用 | 该分组在客户端可见，可编辑可禁用 | 运营创建 / 从禁用恢复 | 已禁用 | 🏢 总后台运营 |
| 分组-已禁用 | 该分组在客户端不可见 | 运营手动禁用 | 已启用 | 🏢 总后台运营 |
| 活动-已发布 | 预设活动在客户端可见，商户可编辑路径 | 运营创建 | 已删除 | 🏢 总后台运营 |
| 活动-已删除 | 预设活动从客户端移除（已配置路径不受影响） | 运营删除 | - | 🏢 总后台运营 |

##### 4.2.4 空值回退规则

推广位路径留空时的行为：

- 开关 OFF（统一模式）：不涉及独立路径，所有推广位直接使用默认推广位路径
- 开关 ON（独立模式）：某个推广位路径留空时，**不视为"未配置"**，而是**自动回退读取"默认推广位路径"的值作为兜底**
- 视觉表现：空值输入框内显示浅灰色提示文字 "默认：/pages/xxx"（xxx 为当前默认推广位路径值），表明该推广位将使用默认路径
- 保存时：空值合法，直接保存空字符串；运行时取值逻辑为 `promoPaths[key] || defaultPath`

> **兜底链**：推广位路径取值优先级：推广位自身路径（非空）> 默认推广位路径 > 空（此时跳转/支付环节校验拦截）

## 模块汇总

### 关键设计决策

1. **预设活动全商户共享、路径独立存储**：预设活动由总后台维护（活动名、AppID、默认推广位路径），所有商户可见同一套列表。每个商户的路径配置（defaultPath 可覆盖 + promoPaths 可按推广位覆盖）独立存储、互不干扰。推广位路径取值采用三级兜底链：推广位自身路径 > 默认推广位路径 > 空。
2. **分组扁平结构**：分组仅一级，不嵌套。客户端按 sortOrder 排列 Tab。
3. **预设活动排序控制客户端展示**：运营通过 sortOrder 排序值控制商户端第三方活动的展示顺序，数字越大越靠前（降序）。
4. **自定义活动归入"其他"**：商户自建活动统一归入"其他"分组，简化分组选择逻辑，降低商户认知负担。
5. **总后台可代客编辑路径**：运营在总后台（B04-D）可直接修改指定客户的第三方活动路径，满足"平台提供推荐路径帮客户填写"的业务场景。
6. **路径空值三级兜底**：推广位路径留空不阻断保存，运行时自动回退读取默认推广位路径；默认路径也为空时，支付/跳转环节再做路径有效性校验。视觉上通过浅灰色提示"默认：/pages/xxx"让商户感知兜底值。

### 待办/注意事项

- 总后台 B04-D 客户路径查阅页面中，需考虑"选择客户"组件的性能（商户量大时）。建议使用远程搜索（输入关键词后才调接口搜索）。
- 预设活动删除前需检查引用商户数，给出提示（"已删除后 N 个商户将失去此活动"）。
